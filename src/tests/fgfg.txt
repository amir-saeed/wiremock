
## Job Parameters

These are passed when triggering the Glue job from the AWS Console:

| Parameter | Example Value | Description |
|-----------|--------------|-------------|
| `JOB_SUBTYPE` | `maintenance` | Fixed value, always `maintenance` |
| `ENV` | `r1c3dev` | Environment name — controls which S3 bucket and Glue catalog DB are used |
| `REPORTING_DATE` | `2026-06-20` | Date being processed (`YYYY-MM-DD`) |
| `EXECUTION_TS` | `20260620120000` | Execution timestamp (`YYYYMMDDHHmmss`) |
| `METASTORE` | `glue` | Metastore type |
| `CONFIG_S3` | `s3://b4c-r1c3dev-eqd-glue-scripts/config/` | S3 path to job config YAML |
| `DAP_S3` | `s3://...` | DAP S3 path |
| `ES_HOST` | `mydb.abc123.eu-west-1.rds.amazonaws.com` | RDS hostname |
| `ES_PORT` | `5432` | RDS port |
| `ES_DATABASE` | `eqd_db` | Database name |
| `ES_USER` | `glue_user` | Database user (IAM auth) |

## Pre-requisites for Testing

Before running the Glue job, you must have source data in the Glue catalog for the reporting date being tested.

### Step 1 — Generate a test parquet file locally

From the `glue/` directory:

```bash
uv run python scripts/generate_parquet_local.py 2026-06-20
```

This generates 15 rows (3 per source: MSM, GOC, CTM, CON, QZN) and prints the exact S3 upload path:

```
✓ File saved locally:
  glue/dist/quotes_2026-06-20.parquet

Upload to S3:
  Bucket : b4c-dal-rc3dev-data-quotes-shredded-lfs
  Folder : quote_journey/shredded_parquet/no_lambda/QUOTE_HEADER/dal_year=2026/dal_month=06/dal_day=20/
  Key    : .../quotes.parquet
```

### Step 2 — Upload parquet to S3

1. Go to AWS S3 Console
2. Navigate to the bucket and folder printed above
3. Upload `glue/dist/quotes_<date>.parquet` as `quotes.parquet`

### Step 3 — Register the partition in Glue Catalog

After uploading, the Glue catalog does not automatically know about the new partition. Run this in Athena:

```sql
MSCK REPAIR TABLE dal_r1c3dev_qj_db.b4c_request_quote;
```

Replace `dal_r1c3dev_qj_db` with the actual Glue catalog database name for your environment.

### Step 4 — Verify data is visible in Athena

```sql
SELECT * FROM dal_r1c3dev_qj_db.b4c_request_quote
WHERE dal_year = 2026 AND dal_month = 6 AND dal_day = 20
LIMIT 5;
```

You should see 15 rows before running the Glue job. If this returns nothing, do not run the job — fix the upload or MSCK REPAIR first.

## Running the Glue Job

1. Go to **AWS Glue Console** → **ETL Jobs** → select `b4c-r1c3{env}-eqd-maintenance`
2. Click **Run**
3. In the **Job parameters** section, set all parameters from the table above
4. Click **Run job**
5. Monitor via **Runs** tab → click the run → **Output logs** (CloudWatch)

---

## Verifying the Run

### Check Glue logs (CloudWatch)

Look for these key log lines in order:

```
[retention] Deleting data older than: <cutoff_date>
[retention] Deleted N quotes older than <cutoff_date>
[partitions] Ensuring partition exists: quote_2026_06
[partitions] Partition created / Partition already exists: quote_2026_06
[ingestion] Ingesting quotes: reporting_date=2026-06-20
[ingestion] Writing quotes: count=15
[ingestion] Quote ingestion complete: rows=15
```

If `count=0` on the ingestion line, no data was read — check Step 3 (MSCK REPAIR TABLE).

### Check PostgreSQL

Connect via pgAdmin or CloudShell psql and run:

```sql
-- Confirm rows landed
SELECT * FROM entity_store.quote
WHERE reporting_date = '2026-06-20';

-- Confirm row count
SELECT COUNT(*) FROM entity_store.quote
WHERE reporting_date = '2026-06-20';
```

Expected: 15 rows with columns `correlationid`, `source`, `reporting_date`, `execution_ts`, `source_timestamp`.

## Test Cases

### TC-01 — Happy Path

**Goal:** Confirm end-to-end flow works for a new date.

1. Upload parquet for a date that has never been processed
2. Run MSCK REPAIR TABLE
3. Run Glue job
4. Verify 15 rows in `entity_store.quote` for that date

**Expected:** Job succeeds, 15 rows in PostgreSQL.

---

### TC-02 — Duplicate Blocked

**Goal:** Confirm the job cannot be run twice for the same date.

1. Run TC-01 successfully
2. Run the Glue job again with the same `REPORTING_DATE`

**Expected:** Job fails with `AlreadyExistsError: data already exists for reporting_date: <date>`.

---

### TC-03 — Retention Deletes Old Records

**Goal:** Confirm records older than 1095 days are deleted.

1. Insert records directly into PostgreSQL with `reporting_date` more than 1095 days ago:

```sql
-- Example: insert a record from 4 years ago
INSERT INTO entity_store.quote (correlationid, source, reporting_date, execution_ts, source_timestamp)
VALUES (
  gen_random_uuid()::text,
  'MSM',
  '2022-01-01',
  '2022-01-01 00:00:00',
  '2022-01-01 00:00:00'
);
```

2. Run Glue job with today's date as `REPORTING_DATE`
3. Check the old records are gone:

```sql
SELECT COUNT(*) FROM entity_store.quote
WHERE reporting_date = '2022-01-01';
```

**Expected:** 0 rows. Deletion logged as `[retention] Deleted N quotes`.

---

### TC-04 — Exactly 1095 Days Old is KEPT

**Goal:** Confirm the boundary — records exactly 1095 days old are not deleted.

1. Calculate the cutoff date: `today - 1095 days`

```python
from datetime import date, timedelta
print(date.today() - timedelta(days=1095))
```

2. Insert a record with `reporting_date` equal to that exact cutoff date
3. Run Glue job
4. Confirm the record still exists

**Expected:** Record is kept. Retention uses `< cutoff_date` (strictly less than), so exactly 1095 days old survives.

---

### TC-05 — New Partition Created

**Goal:** Confirm the job creates a PostgreSQL partition for a month that doesn't exist yet.

1. Use a `REPORTING_DATE` in a future month (e.g. `2026-12-01`)
2. Confirm no partition `quote_2026_12` exists yet:

```sql
SELECT relname FROM pg_class
WHERE relname = 'quote_2026_12';
```

3. Run Glue job
4. Confirm partition was created:

```sql
SELECT relname FROM pg_class
WHERE relname = 'quote_2026_12';
```

**Expected:** Partition `quote_2026_12` is created. Log shows `[partitions] Partition created: quote_2026_12`.

---

### TC-06 — Existing Partition is Not an Error

**Goal:** Confirm the job handles an already-existing partition gracefully.

1. Use a `REPORTING_DATE` in a month whose partition already exists (e.g. `2026-06-25` if `quote_2026_06` was created in TC-01)
2. Run Glue job

**Expected:** Job succeeds. Log shows `[partitions] Partition already exists: quote_2026_06`. No error.

---

### TC-07 — No Parquet in S3

**Goal:** Confirm the job fails clearly when source data is missing.

1. Use a `REPORTING_DATE` for which no parquet was uploaded and no partition registered in the catalog
2. Run Glue job

**Expected:** Job fails with `FileNotFoundError: No parquet data found at path: ...`

## Common Errors and Fixes

| Error | Cause | Fix |
|-------|-------|-----|
| `ModuleNotFoundError: No module named 'contracts'` | Deployed wheel is outdated — missing `contracts/` package | Rebuild wheel with `uv build`, re-upload to S3 |
| `ValueError: 'r1c3dev' is not a valid Env` | `RC3DEV` not in `Env` enum | Add `RC3DEV = "r1c3dev"` to `contracts/enums.py`, rebuild wheel |
| `AlreadyExistsError: data already exists` | Job already ran for this date | Use a different `REPORTING_DATE` or clean up the existing rows first |
| `FileNotFoundError: No parquet data found` | Parquet not uploaded or MSCK REPAIR not run | Upload parquet to S3, then run `MSCK REPAIR TABLE` in Athena |
| `permission denied for schema entity_store` | Glue DB user lacks privileges | DBA to run: `GRANT USAGE ON SCHEMA entity_store TO <user>; GRANT SELECT, INSERT, UPDATE, DELETE ON TABLE entity_store.quote TO <user>;` |
| `count=0` in ingestion logs (job succeeds, no data) | Glue catalog partition not registered | Run `MSCK REPAIR TABLE dal_{env}_qj_db.b4c_request_quote` in Athena |
| `Fatal authentication failed` | Wrong `ES_USER` for IAM auth | Use the same user as the matching job |
| `invalid date '<input variable>'` | `EXECUTION_TS` not set — left as placeholder | Set `EXECUTION_TS` to a real value e.g. `20260620120000` |

## Cleanup After Testing

To re-run the same date, delete the rows first:

```sql
DELETE FROM entity_store.quote
WHERE reporting_date = '2026-06-20';
```

To remove a partition (use carefully):

```sql
DROP TABLE entity_store.quote_2026_06;
```

## Database Access (for testers)

### pgAdmin — generate IAM token via CloudShell

1. AWS Console → CloudShell
2. Run:
```bash
aws rds generate-db-auth-token \
  --hostname <ES_HOST> \
  --port 5432 \
  --username <ES_USER> \
  --region eu-west-1
```
3. Copy the token (valid 15 minutes)
4. In pgAdmin: add server, paste token as password, SSL mode = Require

### Grant permissions (requires admin/master user)

Connect as master user (credentials in AWS Secrets Manager), then:

```sql
GRANT USAGE ON SCHEMA entity_store TO <ES_USER>;
GRANT SELECT, INSERT, UPDATE, DELETE ON TABLE entity_store.quote TO <ES_USER>;
```
